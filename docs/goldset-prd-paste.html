Mini PRD: Goldset Evals

Overview

Goldset Evals is an open source starter kit for evaluating AI-generated marketing work.

The core idea is simple: AI marketing teams do not just need more generation. They need a quality and learning layer that helps them measure whether outputs are good, on-brand, accurate, audience-appropriate, and ready to ship.

The project starts as a CLI-first open source tool, but is designed with a future SaaS path in mind.

Target Users

The first users are early and mid-stage startup teams using AI for growth, marketing, and GTM workflows.

Primary users:

  • Founders
  • Growth marketers
  • Marketing operators
  • Growth engineers
  • Content and lifecycle leads
  • AI-forward GTM teams

These teams are likely using AI to generate lifecycle emails, paid ads, landing page copy, SEO briefs, outbound messages, campaign ideas, and content variants, but they lack a systematic way to evaluate quality.

Problem

AI marketing workflows are easy to scale but hard to trust.

Teams can generate more copy, campaigns, and content than ever, but they do not have a repeatable system for answering:

  • Is this output good?
  • Is it on-brand?
  • Is it accurate?
  • Does it fit the audience and channel?
  • Which parts should be checked by code, by an LLM, or by a human?
  • Are failures recurring?
  • Is the system improving over time?

Without evals, teams either over-automate risky work or keep humans buried in manual review.

Product Thesis

The next moat in AI marketing is not better prompting. It is better evaluation.

Goldset Evals gives marketing teams a way to operationalize judgment through rubrics, deterministic checks, LLM judges, human review, quality reporting, and structured learning loops.

The goal is not to remove marketers from the loop. The goal is to make their judgment compound.

Self-Improving Loop

Goldset Evals should be designed around a simple self-improving loop:

  1. Human review
  2. Structured review signal
  3. Recurring finding
  4. Eval target
  5. Prompt, rubric, workflow, or model improvement
  6. Regression check

This means human review is not just an approval step. It becomes structured evidence that helps the system improve over time.

Example:

  • A marketer reviews an AI-generated lifecycle email and marks it as too generic for founders.
  • That review becomes a structured signal.
  • Several similar signals become a finding: “Lifecycle emails are too generic for founder audiences.”
  • That finding becomes an eval target with regression cases and success criteria.
  • The team improves the prompt, rubric, workflow, or model routing.
  • The eval is rerun to check whether quality improved without causing regressions.

V1 Scope

V1 is an open source CLI tool.

Core capabilities:

  • Load marketing examples from CSV or Google Sheets export.
  • Define quality rubrics in YAML.
  • Run deterministic checks for objective criteria.
  • Run LLM-as-judge checks for qualitative criteria.
  • Store eval runs in SQLite.
  • Generate HTML reports.
  • Provide a local human review UI.
  • Compare human review with machine judgment.
  • Capture human review as structured signals.
  • Group review signals into recurring findings.
  • Export findings as bounded eval targets.
  • Include starter examples for lifecycle email, paid social ads, landing page copy, and SEO/content briefs.

First Use Case

The first fully working example is lifecycle email evaluation.

Example dimensions:

  • Subject line length
  • Body length
  • CTA presence
  • Clarity
  • Audience fit
  • Brand voice
  • Human review required for subjective dimensions

LLM Strategy

OpenAI is the first supported production provider.

The system uses a lightweight provider interface so users can later add other model gateways without changing the core eval system. This could include Anthropic, Gemini, LiteLLM, LangChain, or internal model gateways.

User Workflow

  1. Clone the repo.
  2. Install the CLI locally.
  3. Run evalkit doctor to check setup.
  4. Run the lifecycle email example.
  5. Open the HTML report.
  6. Start the local human review UI.
  7. Submit human judgments, corrections, and failure reasons.
  8. Regenerate the report with calibration metrics.
  9. Run evalkit signals to extract structured review signals.
  10. Run evalkit findings to group repeated failure patterns.
  11. Run evalkit targets to create a bounded improvement task.
